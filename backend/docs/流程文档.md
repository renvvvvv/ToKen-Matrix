## 一、论文拉取分析流程（每日 6-7 天告警设备 → LLM + Search → RAG 归档）

> 目标：每天针对最近 6-7 天出现过告警的设备类型，自动 / 手动拉取相关学术论文，经 LLM 结构化摘要后归档到对应子系统的 RAG 知识库作为源文件。
>
> 涉及子系统：BM（电池）/ EH（动环）/ FR（消防）/ SC（安防）/ BA（楼宇），共 5 大域。

### 1.1 自动版（Auto）

#### 1.1.1 触发方式

- **触发源**：L3 层 "论文分析 · RAG 归档" 模块元信息中明确"每天针对最近 6-7 天有告警的设备类型, LLM + Search 自动拉取论文, 归档为 RAG 源"。
- **触发周期**：每日 1 次（与 L1 巡检数据落库时同步触发，时间窗 14:00 前后，与 `dayPredict` 落库对齐）。
- **关联自动证据**：今日运行日志 (`daylogTimeline`) 中会自动生成 `kind:RAG` 行，例如：
  - `14:30 · BMS 内阻论文归档 · RAG 知识库 · 基于最近 6-7 天告警设备, LLM + Search 拉取论文 2 篇 · 已归档 · Skill: rag/ingest`
- **无需任何手动操作**：本版本在 Hermes Agent 后台按计划任务执行，UI 侧只在"今日运行日志"和"论文流水 & 归档列表"中追加结果条目。

#### 1.1.2 完整细节流程

```
[T+00:00] ① 告警扫描
  · 数据源: PostgreSQL alerts(6-7 天滑窗) + 设备告警关联表
  · 过滤: 5 大系统 BM/EH/FR/SC/BA;排除 P3 巡检类
  · 输出: device_alert_set = {sys, device_id, last_alert_ts, severity, kpi_value}
            ↓
[T+00:30] ② 关键词生成
  · 设备类型 → 关键词映射(系统内置 ontology)
  · 模板: "<device_class> + <kpi_name> + <fault_mode>"
  · 例: BMS-内阻 → ["lithium-ion battery internal resistance evolution",
                     "active cell balancing multi-agent reinforcement learning",
                     "battery SOH estimation impedance threshold"]
            ↓
[T+01:00] ③ 论文检索 (Search: Cloudsway)
  · 数据源: arXiv / IEEE Xplore / Springer / ScienceDirect / ACM
  · 检索方式: Hybrid Search (BM25 + Dense Embedding)
  · 同义词扩展 + 时效性加权 (近 24 月优先)
  · 每关键词返回 12-20 篇候选 → 过滤 2023-2025 → 保留 Top 2-5 高相关度
  · 输出: papers[] = {title, journal, year, confidence, abstract, sys_tag}
            ↓
[T+01:30] ④ LLM 摘要 (Model: claude-opus-4-7 from 小宿科技)
  · 输入: papers[] + device_alert_set (5 类设备工况)
  · 对每篇论文做 3 项抽取:
      ① 关键结论 (1-2 句话)
      ② 实验数据 (数值 / 阈值)
      ③ 适用阈值 (可落地到 BMS/EH/FR/SC/BA 的判断规则)
  · 输出: paper_summaries[] + 设备可落地建议 (按 5 系统归类,通常 12 条)
            ↓
[T+02:30] ⑤ 飞书推送 (真实 webhook)
  · 卡片包含: 报告概况 / 告警设备清单 / Top 5 论文 / 12 项建议
  · 备注: 🔔 由 ToKen Paper Agent 主动触发 · 搜索支持: Cloudsway · LLM: claude-opus-4-7 from 小宿科技
  · HTTP POST → webhook 路由 /webhook/feishu (后端代理避开浏览器 CORS)
            ↓
[T+02:35] ⑥ 等待人工确认
  · 飞书群 @现场负责人(如 @任承智)· 反馈采纳/部分采纳/驳回
  · 兜底: 30 分钟无响应则按"全部采纳"自动归档
            ↓
[T+02:50] ⑦ RAG 归档
  · 文件落库: hermes/domains/{bms|env|fire|security|ba}/rag/{title-slug}.pdf
  · 元数据索引: rag_index.json (含 sys / kpi / confidence / ts)
  · 同步更新 "RAG 归档分布" Xmind 思维导图节点计数 (如 BMS+2 → 40)
  · 同步触发关联 Skill 库条目升级 (如 cell-balancing v1.3 → v1.4)
            ↓
[T+03:00] ⑧ 反馈今日运行日志
  · daylogTimeline 追加 kind:RAG 行
  · RAG 计数 KPI +N
```

## 二、主动巡检展示（告警 → SOP Skill → 自动修复 → 飞书 → 一次链接 → 反馈 → LLM 迭代）

> 目标：演示 Hermes Agent 在 5 大子系统中针对任意告警 (KPI 越阈值) 的 7 步闭环处置能力，闭环率 75%，并展示 LLM 如何基于现场反馈迭代 Skill 版本。
>
> 涉及子系统：BM（电池）/ EH（动环）/ FR（消防）/ SC（安防）/ BA（楼宇），共 5 大域。

### 2.1 自动版（Auto · 每 4h 一次）

#### 2.1.1 触发方式

- **触发源**：L1 层 "跨系统数据透传 · AI 巡检报告" 模块元信息"每 4h 自动巡检 5 大系统参数 → 生成 AI 巡检报告 (6 条/天)"。
- **触发周期**：每 4 小时 1 次，每天 6 次（00:00 / 04:00 / 08:00 / 12:00 / 16:00 / 20:00 / 24:00）。
- **关联自动证据**：
  - "24h AI 巡检趋势" 折柱图 (SVG) 中 13 个时间点数据自动更新。
  - "今日运行日志 (`daylogTimeline`)" 自动追加 `kind:巡检` 行，例如：
    - `04:00 · 凌晨巡检 · AI 综合诊断 · 5 大系统 · 1,280 项 · 巡检得分 96 · 自动巡检报告已生成`
    - `12:00 · 午间巡检 · AI 综合诊断 · 得分 89 · 异常 2 条 · 报告推送`
  - 巡检类型 KPI 实时跳动 (`setInterval 5000ms`，见 `console.html` L2283)。

#### 2.1.2 完整细节流程

```
[T+00:00] ① 周期触发
  · Cron: 每 4h (与 dayPredict 数据对齐)
  · 启动 Hermes Agent 巡检任务 tkm_inspect_{ts}
            ↓
[T+00:01] ② 5 大系统参数全量扫描
  · 总参数: 1,280 项 (每系统约 256 项 KPI)
  · 数据源: SNMP / Modbus / MQTT / BMS 网关
  · 阈值匹配: 与基线阈值表对比
  · 输出: {sys, kpi, value, threshold, severity, ts}
            ↓
[T+00:05] ③ 异常检测 & 分级
  · P0 紧急: 立即告警 + 飞书 + 派单
  · P1 高: 自动触发 Skill SOP
  · P2 中: 触发自动修复
  · P3 低: 巡检留痕
            ↓
[T+00:06] ④ 触发 Skill SOP
  · 从 125+ Skills 中匹配主修复 Skill (Hermes 域匹配)
  · 例: BMS 内阻越限 → 主 Skill bms/internal-r, 辅助 5 个
  · 执行: 自动下发修复指令 / 调整控制参数
            ↓
[T+00:08] ⑤ KPI 实时透传
  · 5 大系统 KPI 视图同步刷新
  · 异常参数标记为红色 alert 状态
  · 修复成功后翻转为绿色 fix 状态
            ↓
[T+00:09] ⑥ 飞书推送
  · 卡片包含: 巡检得分 / 异常条目 / 自动闭环率 / Skill 执行清单
  · 走 webhook 真实推送
            ↓
[T+00:10] ⑦ 报告归档
  · 巡检报告写入 ai_report_{date}.json
  · daylogTimeline 追加巡检/告警/收敛/迭代/RAG 行
  · 触发 paperPullAuto (若发现新设备告警模式)
```

## 四、附录：常见排查

1. **看不到弹窗**：确认已进入 `Agent 运营管理平台` 页面 (A0 菜单)，且 5 大 L 层级都已展开 (L1/L2/L3 都有 `.section-title`)。
2. **飞书推送失败**：弹窗底部会显示 `⚠️ webhook 调用失败 · err`，可点击「复制 curl」用服务端命令手动重发。
3. **LLM 迭代超时**：`claude-opus-4-7 from 小宿科技` 离线时自动降级到兜底规则"基于反馈调整 → 增加联动"，并仍会生成 Skill 版本升级记录。
4. **计时器不归零**：关闭弹窗会停止计时，再次打开会重新从 00:00:00 开始。
5. **模型名 / 搜索名对外显示**：
   - 前端全部展示 `claude-opus-4-7 from 小宿科技` / `Cloudsway`；
   - 实际后端调用仍走 `MiniMax-M2.7-highspeed` 与 `https://api.minimaxi.com/v1`，不要在浏览器控制台 Network 中看到 `MiniMax-M2.7-highspeed` 字样就以为"对外名没生效"——这只是真实请求体中的 model 字段，对外一律以 UI 文案为准。
